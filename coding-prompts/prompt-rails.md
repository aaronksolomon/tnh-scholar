# Prompt Rails

Write prompts for a strong engineering peer, not for a novice.

Assume the agent:

- reads repo `AGENTS.md` first
- can inspect the codebase directly
- can infer routine engineering steps without hand-holding
- can make local tradeoffs if the objective and boundaries are clear

## Core Rules

1. State the objective in the first sentence.
2. Name the hard constraints and success conditions explicitly.
3. Point to the few files, commands, or docs that actually matter.
4. Leave routine execution details implicit unless they are unusual or safety-critical.
5. Prefer bounded scope over narrated background.
6. Ask for outputs that help review: changed files, brief summary, open questions.

## Default Structure

Use this shape unless the task clearly needs something else:

```md
# <Task Title>

Objective: <one clear sentence>

Context:
- <only facts that materially affect the work>

Requirements:
- <hard requirement>
- <hard requirement>

Scope:
- in: <expected files or surfaces>
- out: <explicit exclusions>

Validation:
- <tests/checks if they matter>

Deliverable:
- <what to return at the end>
```

## What To Avoid

- telling the agent to read `AGENTS.md` or repo standards when that is already assumed
- long motivational framing
- step-by-step instructions for obvious engineering work
- repeating repo context the agent can inspect faster than it can read
- vague verbs like "improve" without acceptance criteria
- excessive guardrails that weaken initiative

## When More Detail Is Worth It

Add detail only when one of these is true:

- the task has unusual safety constraints
- there is a narrow architectural boundary to preserve
- success depends on a specific precedence rule, invariant, or output shape
- the task is exploratory and the evaluation frame matters as much as the code

## Tone

- direct
- technical
- sparse
- peer-level

## Good Pattern

```md
# tnh-conductor docs cleanup

Objective: tighten the maintained `tnh-conductor` docs for release readiness.

Context:
- the maintained operator surface is `run`, `status`, and `status --watch`
- `tnh-conductor-spike` and `conductor_mvp` should stay clearly demoted

Requirements:
- add or refine operator-facing docs only
- update discoverability where needed
- avoid broad architecture rewrites

Scope:
- in: `docs/cli-reference/`, `docs/development/`, relevant landing pages
- out: unrelated README churn, autogenerated maps unless required for discoverability

Deliverable:
- changed files
- short summary
- residual doc gaps
```

## Bad Pattern

```md
Read all the repo instructions carefully. Think step by step. First inspect the repository, then inspect the docs, then make a plan, then update files carefully, then summarize everything in detail. Be concise but also comprehensive.
```
